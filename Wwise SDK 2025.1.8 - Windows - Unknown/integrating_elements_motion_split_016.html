Support for Meta Quest through OpenXR

Wwise Motion supports haptic feedback on Meta Quest Touch controllers through the OpenXR API, using the XR_FB_haptic_pcm extension. Wwise Motion Meta Quest support is available on Android (Meta Quest standalone) and on Windows when using Meta Horizon Link. The Motion sink accepts a Stereo 2-channel waveform. Channel 0 drives the left controller and channel 1 drives the right controller.

Note: For Unity and Unreal Engine projects, the Wwise integration packages for those engines already handle the AkMotionSink linking. For Meta Quest OpenXR haptics setup in those environments, see Using Unity OpenXR for Haptic Feedback on Meta Quest or Using Unreal OpenXR for VR Haptic Feedback on Meta Quest respectively.

Requirements

  • The XR_FB_haptic_pcm extension is required. This extension is only supported on the Meta OpenXR runtime.
  • This feature has been validated on Meta Quest Touch Plus Controllers. Meta Quest Touch Controllers and Meta Quest Touch Pro Controllers are expected to be supported but have not been tested.

Because AkMotionSink does not link directly to the OpenXR loader, you must pass your application's OpenXR function pointers and handles to the Motion plug-in before calling AK::SoundEngine::AddOutput. To do this, include the following header:

#include <AK/Plugin/AkMotionSinkOpenXRHelpers.h>

You also need a haptic XrAction in your OpenXR action set, bound to the /user/hand/left/output/haptic and /user/hand/right/output/haptic paths. When your XrInstance, XrSession, and haptic XrAction handles are ready, provide them to the Motion plug-in:

  • If AkMotionSink is integrated as a static library, call AKMOTIONSINK_SET_OPENXR_FUNCTIONS_AND_HANDLES_WWISE_STATIC inside a function, after the handles are available:
    AKMOTIONSINK_SET_OPENXR_FUNCTIONS_AND_HANDLES_WWISE_STATIC(xrInstance, xrSession, xrHapticAction);
  • If AkMotionSink is integrated as a dynamic library, call AKMOTIONSINK_SET_OPENXR_FUNCTIONS_AND_HANDLES_WWISE_DYNAMIC after Init.bnk is loaded and AkMotion.dll (or libAkMotion.so) is initialized:
    AKMOTIONSINK_SET_OPENXR_FUNCTIONS_AND_HANDLES_WWISE_DYNAMIC(xrInstance, xrSession, xrHapticAction);

After this call, add a Motion output with a device ID of 0 as usual:

AkOutputSettings outputSettings("Wwise_Motion", 0);
AK::SoundEngine::AddOutput(outputSettings);
Note:
  • On Android, if AKMOTIONSINK_SET_OPENXR_FUNCTIONS_AND_HANDLES_WWISE_STATIC or AKMOTIONSINK_SET_OPENXR_FUNCTIONS_AND_HANDLES_WWISE_DYNAMIC is not called before AddOutput, the sink automatically falls back to the Android device's built-in vibration.
  • On Windows, when OpenXR functions have been initialized, a device ID of 0 routes to the OpenXR haptic sink instead of the default XInput device.
